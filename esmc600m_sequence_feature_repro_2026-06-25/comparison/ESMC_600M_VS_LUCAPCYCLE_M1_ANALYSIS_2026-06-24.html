ESM-C 600M 与 LucaPCycle M1 特征技术差异分析

生成日期：2026-06-24。分析对象：EnzymeCAGE 本地 ESM-C 序列特征代码、LucaPCycle V3 源码、Nature Communications 论文方法部分。

核心结论

不等价如果你的目标是复现论文中的 M1，现有 ESM-C 600M 特征不能直接当作 LucaPCycle 的 M1 使用。

原因不是保存格式或 pooling 细节，而是上游蛋白语言模型不同：我们本地用的是 esmc_600m，LucaPCycle M1 用的是 esm2_t36_3B_UR50D。

实施建议

建议分两步第一步先实现并接入论文的 M2，因为这是我们当前没有的新特征类别。

第二步做消融：GVP + ESM-C600M + M2 作为低成本方案；若需要严格对齐 LucaPCycle，再补提 ESM2-3B M1。

论文方法要点

Nature 论文把 LucaPCycle 描述为双通道模型：输入序列同时做 character-level 和 word-level tokenization。character-level 通道使用 ESM2-3B 产生表示矩阵 M1，word-level 通道使用 BPE 分词后交给 Transformer-Encoder 产生 M2。Pooler 再用 Value-Level Attention Pooling 把矩阵降采样成向量，最后拼接两个 pooled 向量做分类。

论文还明确说明：第一个通道为 ESM2-3B contextual representation；第二个通道为 4 层、8 头 Transformer-Encoder。源码中的 V3 训练脚本与这点一致。

本地 EnzymeCAGE ESM-C 600M 做了什么

本地主要代码位于：

  • custom/data_build/run_esmc_sequence_only.py
  • feature/main.py 中的 calc_seq_esm_C_feature

关键实现：

model = ESMC.from_pretrained("esmc_600m").to(device)
protein = ESMProtein(sequence=seq)
protein_tensor = model.encode(protein)
logits_output = model.logits(
    protein_tensor,
    LogitsConfig(sequence=True, return_embeddings=True),
)
node_feature = logits_output.embeddings[0].cpu().numpy()
seq_to_feature[seq] = node_feature.mean(axis=0)

保存结果有两种粒度：

  • node_level/{UniprotID}.npz：每个残基一个向量，已知维度为 seq_len x 1152。
  • protein_level/seq2feature.pkl：对全序列 residue matrix 做 mean pooling，得到 1152 维全局向量。

LucaPCycle M1 源码做了什么

我没有在当前 Linux 工作区找到你说的 Windows 路径 F:\CHR\LucaPCycle\LucaPCycle-3\LucaPCycle-3，所以临时拉取了公开仓库 https://github.com/LucaOne/LucaPCycle.git 的 V3 分支，commit 为 d1633791574661ef82486ccc192fe0d9aef00b3e。

源码中 M1 的 embedding 生成在 src/llm/esm/predict_embedding.py，关键实现：

elif version == "3B":
    llm_name = "esm2_t36_3B_UR50D"
    global_layer_size = 36
    global_model, global_alphabet = pretrained.load_model_and_alphabet(
        "esm2_t36_3B_UR50D"
    )

out = global_model(tokens, repr_layers=repr_layers, return_contacts=False)
embedding = out["representations"][global_layer_size] \
    .to(device="cpu")[0, 1: truncate_len + 1].clone().numpy()

V3 训练脚本进一步确认：

embedding_input_size=2560
matrix_max_length=3072
MATRIX_POOLING_TYPE="value_attention"
llm_version="esm2"
llm_type="esm"
llm_step="3B"

因此，LucaPCycle 的 M1 是 ESM2-3B 最后一层 residue representation matrix，训练时以 2560 维输入进入 matrix channel，并通过 value_attention pooling 降维后再与 M2 pooled vector 拼接。

是否需要二次提取 M1

如果目标是严格复现或借鉴 LucaPCycle 的 M1：需要二次提取 ESM2-3B 的 residue matrix。现有 ESM-C 600M 不能当作等价 M1，因为维度、模型、训练接口、pooling 输入都不一致。

如果目标是工程上快速扩充酶特征：可以暂时不重提 M1，先把现有 ESM-C 600M 作为“已有 PLM residue/sequence feature”，再新增 LucaPCycle 的 M2 word-level Transformer 特征。这样能先回答新特征类别是否带来增益。

我建议的实验路线

  1. 先做 M2。实现 BPE word-level tokenizer，复用 LucaPCycle 的 subword vocab/codes 或在我们的酶序列上训练同规模 BPE，然后用 4 层 8 头 Transformer-Encoder 输出 M2 pooled vector。
  2. 先跑低成本组合。训练 GVP + ESM-C600M + M2，与当前 GVP + ESM-C600M 比较。
  3. 若 M2 有增益，再决定是否补 M1。补提 ESM2-3B M1 成本高，但可以做 GVP + ESM2-3B-M1 + M2 与 GVP + ESM-C600M + M2 的直接对照。
  4. 不要把 ESM-C 600M 文件命名为 M1。建议在代码和结果表中明确叫 ESMC600M 或 existing_plm，避免后续论文/汇报时混淆。

一句话决策

不要把已提取的 ESM-C 600M 视为 LucaPCycle M1 的等价替代。但在当前项目里，它仍然是有价值的 PLM 序列特征。为了控制算力成本，最合理的下一步是新增并评估 M2，再用消融结果决定是否补提 ESM2-3B M1。
