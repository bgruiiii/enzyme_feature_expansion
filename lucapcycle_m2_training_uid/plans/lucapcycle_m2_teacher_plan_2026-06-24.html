LucaPCycle M2 酶序列特征提取方案

日期：2026-06-24 | 当前目标：先提取酶的 word-level M2 特征，后续与 GVP、ESM-C 600M/ESM-pocket 一起训练新的酶检索模型。

1. 方案目的

我们计划在原 EnzymeCAGE 酶检索模型已有的结构特征和序列特征之外，新增 LucaPCycle 论文中的 M2 表征。 M2 来自氨基酸序列的字词级切分和 Transformer 编码，可以作为一种独立的酶序列语义特征。

本阶段只做 M2 特征提取和质控，不直接开展最终训练。等 M2 在 HPC 上完整提取并和 UniProtID 对齐后，再进入新一轮 EnzymeCAGE-style 检索模型训练。

2. M1 与 M2 的区别

名称 输入粒度 论文原始来源 我们项目中的处理 当前分工
M1 character-level，即单个氨基酸字符 ESM2-3B 蛋白语言模型 不采用 3B 模型，直接用项目早期已经在 HPC 提取好的 ESM600M/ESM-C 600M 序列特征替代 已有数据，不在本阶段重新提取
M2 word-level，即 BPE/subword 字词级片段 LucaPCycle 训练好的 Transformer-Encoder 分支 沿用 LucaPCycle 的 BPE 词表、BPE codes 和训练好的 M2 encoder checkpoint 当前由我们负责

论文 Methods 中说明：Tokenizer 会同时进行 character-level 和 word-level tokenization； Encoder 中一路用 ESM2-3B 得到 M1，另一路用 Transformer-Encoder 从 word-level list 得到 M2。 因此 M2 不是 ESM 特征，也不是我们已有的 ESM-C 600M 特征。

与论文不同的一点：论文的 M1 使用 ESM2-3B；我们项目中用早期已经提取并存放在 HPC 的 ESM600M/ESM-C 600M 序列特征来替代。 因此当前不需要重新提取 M1，只需要完成 M2 提取并和 HPC 现有特征按 UniProtID 对齐。

计划保存的特征

特征名 形状 用途
m2_token_matrix_fp16 (BPE token 数, 1024) 最接近论文中 M2 的 token-level 表征矩阵，体积较大，作为可选审计产物
m2_mean (1024,) 对 M2 token 矩阵按 attention mask 做平均池化，作为训练主特征之一
m2_max (1024,) 对 M2 token 矩阵做 mask-aware 最大池化，补充局部强信号
m2_value_attention (1024,) 使用 LucaPCycle 原模型的 value_attention 池化结果
m2_projected_256 (256,) 经过 LucaPCycle seq_linear 后的低维向量，可作为轻量融合候选

4. 技术路线

氨基酸序列
  -> 使用 LucaPCycle 的 BPE codes 做 word-level 切分
  -> 使用 LucaPCycle tokenizer 转成 input_ids 和 attention_mask
  -> 加载 LucaPCycle V3 训练好的 checkpoint
  -> 调用模型内部 seq_encoder，即 Transformer-Encoder 分支
  -> 取 seq_outputs[0] 作为 M2 token 矩阵
  -> 池化得到 m2_mean / m2_max / m2_value_attention / m2_projected_256
  -> 按 UniProtID 分片保存，供后续 EnzymeCAGE 训练读取

这里必须加载 checkpoint。原因是 BPE 只负责把氨基酸序列切成 token； 真正把 token 编码成 M2 向量的是 Transformer-Encoder 的参数，这些参数来自 LucaPCycle 训练好的权重。 没有 checkpoint，就只能得到 token ID，得不到论文意义上的 M2 表征。

5. 代码依据

环节 本地源码位置 作用
BPE 切词 src/inference.py, src/batch_converter.py 使用 BPE(bpe_codes, merges=-1, separator='') 对序列做 subword 切分
BPE codes subword/extra_p/extra_p_50_codes_20000.txt LucaPCycle V3 使用的字词级合并规则
BPE vocab vocab/extra_p/extra_p_50_subword_vocab_20000.txt word-level token 到 ID 的词表
模型类 src/lucaprot/models/lucaprot.py 定义 LucaProt 双通道模型
M2 编码器 self.seq_encoder = BertModel(...) LucaPCycle 内部训练的 Transformer-Encoder 分支
M2 矩阵 seq_outputs = self.seq_encoder(...) 后的 seq_outputs[0] 每个 BPE token 的 1024 维上下文表示

现成的 prediction_v2.py 主要用于分类预测结果输出，不是专门导出 M2。 因此我们会写一个小的自定义提取脚本，复用 LucaPCycle 的 tokenizer、checkpoint 和 seq_encoder。

7. HPC 执行计划

  1. 把 LucaPCycle V3 代码和 TrainedCheckPoint (1).tar.gz 上传到 HPC，优先上传压缩包并在 HPC 端解压，便于校验完整性。
  2. 在 HPC 端记录 md5sum，检查 subword/、vocab/、models/ 和关键 checkpoint 文件是否齐全。
  3. 使用已建立的环境：/public/home/acfbwjsi7s/envs/lucapcycle_m2，在计算节点加载 compiler/rocm/dtk-23.10 后运行。
  4. 先抽取 10 个 UniProtID 做样例，验证 checkpoint 能加载、输出维度为 1024、无 NaN/Inf、UID 与序列 hash 能对齐。
  5. 样例通过后，对 107,731 个训练 UniProtID 分片提取 M2，输出 shard manifest 和 failed UID 表。
  6. 最后检查 145,607 条训练样本能否全部按 UniProtID 映射到 M2 特征。

8. 质控标准

  • 输入 UID 数、成功 UID 数、失败 UID 数都有记录。
  • 每个输出保存 UniprotID、sequence_sha256、sequence_length、bpe_token_count。
  • m2_mean、m2_max、m2_value_attention 的维度应为 1024。
  • m2_projected_256 的维度应为 256。
  • 所有向量检查无 NaN/Inf。
  • 池化时使用 attention mask，避免 padding token 污染结果。
  • 所有分片输出都写入 manifest，便于训练阶段按 UID 追踪。

9. 与后续模型训练的关系

M2 提取完成后，会作为新增酶侧特征流，与原有 GVP 结构/口袋特征、HPC 现有 ESM600M/ESM-C 600M/ESM-pocket 序列特征一起进入新的 EnzymeCAGE-style 酶检索模型训练。 由于输入特征维度和模态增加，后续训练阶段需要相应调整特征融合层、投影维度、batch size、学习率和消融实验设计。

这里需要特别说明：我们不是完全照搬 LucaPCycle 的双通道模型做训练，而是借用其 M2 特征提取思路。 后续检索模型仍采用我们原 EnzymeCAGE 的训练框架，只是在酶侧特征中新增 M2，并把论文 M1 位置替换为已经在 HPC 上存在的 ESM600M/ESM-C 600M 序列特征。

微生物分支暂时不进入本阶段。大计划中，后续会在酶检索之后加入微生物对底物代谢偏好的检索打分，并把降解通路预测作为前置模块。
